图片展示页面制作笔记（一）

做了一个储存设计素材的页面。
其中只展示指定数量的图片的代码和可以检测文件夹里所有图片并显示出来的代码都很有学习意义。下面先讲展示指定数量图片的代码。

JavaScript基础知识部分

1.querySelector()方法

document.querySelector()方法可以选择文档中符合CSS选择器规则的第一个元素。
例如：

document.querySelector(“p”);

可以选择文档中第一个p元素。

document.querySelector(“.myclass”);

可以选择文档中第一个class为”myclass”的元素。

document.querySelector(“#myid”);

可以选择文档中第一个id为”myid”的元素。

document.querySelectorAll()方法可以选择文档中符合CSS选择器规则的所有元素。

基本语法

const elements = document.querySelectorAll(selector);

参数说明

selector：一个字符串，包含一个或多个CSS选择器
返回值：一个NodeList集合（类似数组但不是数组）

但是，使用querySelectorAll()方法时，后面需要用forEach()方法遍历集合中的元素。
例如：

<script>document.querySelectorAll("p").style.backgroundColor = "red";</script>

是无法将所有p元素的背景颜色设置为红色的。

正确的写法应该是：

<script>document.querySelectorAll("p").forEach(p => p.style.backgroundColor = "red")</script>

检测图片是否存在的函数

为什么需要这个函数呢？因为图片有可能是各种不同格式的。
代码片段如下

//检测图片是否存在的函数
async function checkImageExists(url) {
return new Promise((resolve, reject) => {
const img = new Image();
img.onload = () => {
resolve(true);
};
img.onerror = () => {
resolve(false);
};
img.src = url;
});
};

为什么要使用异步函数？

  1. 图片加载是异步操作

因为图片的加载是异步的，如果直接使用同步的方式，会导致页面卡顿。
2. 需要等待加载结果
需要异步等待结果

代码解析

async function checkImageExists(url) {
    // 1. 创建 Promise
    return new Promise((resolve, reject) => {
        // 2. 创建图片对象
        const img = new Image();
        

    // 3. 设置成功回调（未来某个时间执行）
    img.onload = () => {
        console.log('图片加载成功！');
        resolve(true); // 这里会在图片加载完成后执行
    };

        resolve(false); // 这里会在图片加载失败后执行
    };

// Promise 立即返回，但结果要等回调执行



}

如果不使用异步函数

点击展开同步版本的代码 

// ❌ 同步版本（无法工作）
function checkImageExistsSync(url) {
    const img = new Image();
    let exists = false;
    

img.onload = () => {
    exists = true; // 这个回调会在未来执行
};

img.onerror = () => {
    exists = false; // 这个回调会在未来执行  
};

img.src = url;

return exists; // 这里总是返回 false！
// 因为回调函数还没执行！



}


// 测试
const result = checkImageExistsSync(‘https://example.com/image.jpg‘);
console.log(result); // 总是 false，即使图片存在

同步代码失败的原因:

点击展开同步版本的代码 

function checkImageExistsSync(url) {
    const img = new Image();
    let exists = false;  // 第1步：初始化为false
    

img.onload = () => {
    exists = true;   // 第4步：回调执行（但这时函数已经返回了！）
};

img.onerror = () => {
    exists = false;  // 第4步：回调执行（但这时函数已经返回了！）
};

img.src = url;       // 第2步：开始异步加载

return exists;       // 第3步：立即返回false
// ↑ 这里返回时，onload/onerror回调还没执行！



}

同步版本的时间线是这样的

点击这里查看代码解析 

console.log('1. 调用同步函数');
const result = checkImageExistsSync('image.jpg');
console.log('2. 得到结果:', result); // 总是false
console.log('3. 一段时间后...');
// 这时候onload/onerror回调才可能执行，但已经太晚了！


// 输出：
// 1. 调用同步函数
// 2. 得到结果: false
// 3. 一段时间后…
// [如果图片存在] 图片加载完成回调执行

异步版本的时间线是这样的

点击这里查看代码解析

console.log('1. 调用异步函数');
checkImageExists('image.jpg')
    .then(result => {
        console.log('3. 得到结果:', result); // 正确的true/false
    });
console.log('2. 继续执行其他代码');


// 输出：
// 1. 调用异步函数
// 2. 继续执行其他代码
// [等待图片加载…]
// 3. 得到结果: true 或 false

可见，同步版本将总是按照从上到下的顺序来执行，而上面的回调函数由于需要等待图片加载结果才能执行。同步版本失败就是因为代码按顺序执行，等不到异步回调完成就返回了。即，根本原因是：同步代码无法等待异步操作完成，因为：

  1. 单线程特性：JavaScript是单线程的，不能”暂停”等待

  2. 非阻塞设计：如果同步等待，整个页面会卡住

  3. 事件驱动：异步操作完成后通过回调通知

关于“未来某个时间执行”

关键要点

  • 不是批量完成：每个图片独立加载，独立完成

  • 并行加载：浏览器会同时发起多个网络请求

  • 完成顺序不确定：取决于文件大小、网络、服务器等

  • “未来” = 该图片加载完成时：可能是几毫秒，也可能是几秒后

当检测26张图片时，浏览器会同时发起多个请求，每张图片在自己的加载完成后立即触发回调，而不是等待所有图片都加载完成！

总结

使用异步函数的原因是：

  1. 图片加载本质是异步的 - 需要时间，结果不是立即可得

  2. 需要等待操作完成 - 才能知道图片是否存在

  3. 避免阻塞主线程 - 让浏览器可以同时处理其他任务

  4. 代码可读性 - async/await 让异步代码看起来像同步代码

拓展

什么时候需要使用async/await

  1. 当你有连续的异步操作时（避免嵌套越来越深的回调地狱）
  2. 当需要基于前一个结果做判断时
  3. 当错误处理需要同步风格时
  4. 有复杂的异步逻辑流程

问自己一些问题：

  1. “我需要等待某个操作完成吗？”

是 → 可能需要 async/await

  1. “后续代码依赖这个异步结果吗？”

是 → 可能需要 async/await

  1. “我有多个需要顺序执行的异步操作吗？”

是 → 可能需要 async/await

  1. “我只是启动一些并行操作，然后等所有完成？”

是 → 用 Promise.all()，不一定需要 async/await

  1. “我只有一个简单的异步操作？”

是 → 直接用 Promise，不需要 async/await

也就是说

  • 后面函数需要等待前面函数完成 → 在等待的操作前写 await

  • 函数内部使用了await→ 在该函数前写async

在我的代码中的依赖关系如下所示：

checkImageExists(url) ← 基础层（返回 Promise）
↓ 等待完成
getImageURL(number) ← 中间层（async/await）
↓ 等待完成
createImageGallery() ← 业务层（async/await）
↓ 等待完成
displayGallery() ← 展示层（async/await）

当函数B需要等待函数A完成才能继续时：

在函数B内部，在调用函数A的地方前面加await

在函数B本身前面加async

更清晰的判断规则
规则1：函数内部有 await → 函数前加 async
规则2：需要等待其他函数完成 → 前面加 await
规则3：没有依赖关系 → 不需要 await

原因：

  1. 设计意图：检测功能VS确保功能

 return new Promise((resolve) => {
     const img = new Image();
     img.onload = () => resolve(true);   // 存在 → 正常结果

 });

// 如果使用 reject：回答”图片必须存在”
async function ensureImageExists(url) {
return new Promise((resolve, reject) => {
const img = new Image();
img.onload = () => resolve(true); // 成功
img.onerror = () => reject(new Error(‘图片不存在’)); // 失败
img.src = url;
});
}

最佳实践总结

使用 resolve 的情况：

  • 所有可能的结果都是有效业务状态

  • 调用者需要处理多种正常情况

  • “失败”不是异常，而是预期内的结果

使用 reject 的情况：

  • 真正的错误或异常情况

  • 参数验证失败

  • 需要中断流程的严重问题

思考：此处无法加载图片是一种失败结果吗？或者说我们应该将它当作一种失败结果吗？

关于async/await和.then()/.catch()

点击这里查看代码解析

// 这两个函数是等价的：


// async/await 版本
async function asyncVersion() {
    try {
        const a = await task1();
        const b = await task2(a);
        return b;
    } catch (error) {
        console.error(error);
        return null;
    }
}


// .then()/.catch() 版本
function thenVersion() {
    return task1()
        .then(a => task2(a))
        .catch(error => {
            console.error(error);
            return null;
        });
}

关于checkImageExists()函数返回值的问题

思考：checkImageExists()函数的返回值，究竟是return new Promise()，即返回一个Promise对象，还是返回resolve中传递的true和false？

正确答案：返回的是 Promise 对象

点击这里查看代码解析

async function checkImageExists(url) {
    return new Promise((resolve, reject) => {
        const img = new Image();
        img.onload = () => {
            resolve(true);  // ← 这设置的是 Promise 的"结果值"
        };
        img.onerror = () => {
            resolve(false); // ← 这设置的是 Promise 的"结果值"  
        };
        img.src = url;
    });
    // ↑ 这里返回的是 Promise 对象本身，不是 true/false
}

类比理解
就像是一个”礼物盒”的比喻：

点击这里查看代码解析

// 函数返回的是"礼物盒"(Promise)，不是"礼物"(true/false)
function giveGift() {
    return new Promise((deliverGift) => {
        setTimeout(() => {
            deliverGift('一块手表'); // 往盒子里放礼物
        }, 1000);
    });
    // 立即返回的是空盒子，不是手表
}


const giftBox = giveGift(); // 得到空盒子(Promise)
// 稍后打开盒子才能拿到手表(true/false)
const actualGift = await giftBox; // 得到手表

#### 为什么这样设计？ 同步函数 vs 异步函数
点击这里查看代码解析

// 同步函数：直接返回值
function syncFunction() {
    return true; // ← 立即返回 true
}


// 异步函数：返回”值的容器”
function asyncFunction() {
    return new Promise((resolve) => {
        setTimeout(() => {
            resolve(true); // ← 设置容器的内容
        }, 1000);
    }); // ← 立即返回容器(Promise)
}

总结

  • 函数返回的是：Promise对象

  • resolve传递的是：Promise对象的”结果值”（true/false）

两者关系是：Promise是容器，true/false是容器里的内容

所以当您调用 checkImageExists() 时，得到的是一个”装有布尔值的盒子”，而不是布尔值本身。需要打开盒子（用await或.then()）才能拿到里面的true/false。

是否可以这样理解：checkImageExists函数调用时，Promise被立刻调用和返回，此时Promise对象里还没有内容；当得出图片加载结果（成功或者失败）时，resolve（true）或resolve（false）执行，Promise的状态被更改，同时Promise对象里被装入true或false。

完全正确！
